\begin{frame}{¿En qué consiste la programación dinámica?}
  \begin{itemize}
   \item Es una técnica de solución de problemas \textbf{recursiva}.
   \item Al igual que DyC, se descompone en \textbf{subproblemas más pequeños de la
   misma especie}, para resolverlos recursivamente y combinar esas soluciones.
   \item La \textbf{diferencia} importante:
         \begin{itemize}
            \item En DyC, los subproblemas son independientes entre sí, y se resuelven individualmente.
            \item PD es aplicable cuando los subproblemas \textbf{no son independientes}.
         \end{itemize}
   \item De realizar una implementación recursiva directa, se \textbf{recalcularían} los mismos subproblemas muchas veces.
  \end{itemize}
  {\hfill \includegraphics[scale=0.3]{captionfibotree.png} \hfill}
    
\end{frame}

\begin{frame}{¿En qué consiste la programación dinámica? (2)}
  \begin{itemize}
   \item Solución: \textbf{almacenar} las soluciones a subproblemas.
   \item De esta forma cada subproblema se calcula \textbf{una sola vez}.
   \item Se podría decir muy simplificadamente que: DyC $+$ Caching = PD
   \item Es común usar PD para problemas de \textbf{optimización}:
       \begin{itemize}
           \item Encontrar una solución que \textbf{minimice} cierta función objetivo, en un espacio de soluciones posibles.
           \item Como la técnica es recursiva, para poderse aplicar el problema debe cumplir el \textbf{principio del óptimo}.
       \end{itemize}
   \item Principio del óptimo:
       \begin{itemize}
            \item \textbf{Las partes de una solución óptima} a un problema, deben ser \textbf{soluciones óptimas de los correspondientes subproblemas}.
            \item Es lo que permite obtener una solución óptima al problema original a partir de soluciones óptimas de los subproblemas.
       \end{itemize}
  \end{itemize}
    
\end{frame}

\begin{frame}{El esquema general [Según CLRS]}
    Los algoritmos de programación dinámica se pueden organizar típicamente en 4 pasos que responden al siguiente esquema general:
   \begin{enumerate}
   \item Caracterizar la estructura de una solución óptima.
   \item Definir recursivamente el valor de una solución óptima.
   \item Computar el \textbf{valor} de una solución óptima. Se calcula de manera \textit{bottom-up}. [Según CLRS: Puede ser top-down en AED 3]
   \item Construir una solución óptima a partir de la información obtenida en el paso 3
  \end{enumerate}
    El paso 4 es optativo: se omite si solo interesa el valor o puntaje.
\end{frame}

\begin{frame}{Problema del recorrido óptimo en una matriz}

\small

	Ejercicio 3.8, práctica 3 de Algoritmos y Estructuras de Datos 3:

\ 

	Sea $M \in \mathbb{N}^{m \times n}$ una matriz de números naturales. Se desea obtener un camino que empiece en la casilla superior
    izquierda $(1,1)$, termine en la casilla inferior derecha $(m,n)$ y tal que minimice la suma de los valores de las casillas por
    las que pasa. En cada casilla $(i,j)$ hay dos movimientos posibles: ir hacia abajo (a la casilla $(i+1, j)$), o ir hacia la
    derecha (a la casilla $(i, j+1)$).
   \begin{itemize}
   \item[a] Diseñar un algoritmo eficiente basado en programación dinámica que resuelva este problema.
   \item[b] Determinar la complejidad del algoritmo propuesto (temporal y espacial).
   \item[c] Exhibir el comportamiento del algoritmo sobre la matriz que aparece a continuación.
  \end{itemize}
                \[ \left [  \begin{array}{cccc}
2 & 8 & 3 & 4 \\
5 & 3 & 4 & 5 \\
1 & 2 & 2 & 1 \\
3 & 4 & 6 & 5 \\
 \end{array} \right ] \]

\end{frame}

\begin{frame}{Relevancia del ejercicio elegido}
    \begin{itemize}
        \item Patrón de recursión sencillo (solo 2 llamadas), pero no trivial.
        \item Los subproblemas se pueden ``ver'' en la misma matriz.
        \item Camino mínimo en un DAG (reveer el ejercicio más adelante).
        \item Pide complejidad y reconstrucción del camino: es un ejercio completo, que ejercita todos los pasos generales de PD.
        \item Por todo esto, es un buen ejercicio para usar como primer problema de optimización en el que los subproblemas se definan con 2 variables ($i$,$j$), luego
                de haber visto ejemplos de PD unidimensionales.
    \end{itemize}
     
    
\end{frame}


\begin{frame}{Fórmula recursiva}
    \small
    La matriz de resultados parciales almacena en $best(i,j)$ la mínima longitud de un camino que empiece en $(i,j)$ y llegue
    a $(m,n)$, haciendo solo movimientos hacia abajo y hacia la derecha.
   \begin{itemize}
   \item $best(i,j) = M_{i,j} + min(best(i+1,j), best(i,j+1))$ \\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ para $1 \leq i < m$ y $1 \leq j < n$
   \item $best(i,n) = M_{i,n} + best(i+1,n)$ para $1 \leq i < m$
   \item $best(m,j) = M_{m,j} + best(m,j+1)$ para $1 \leq j < n$
   \item $best(m,n) = M_{m,n}$
  \end{itemize}
    La longitud del mínimo camino entre esquinas, que constituye la solución al problema, viene dada por $best(1,1)$. Con esto ya podríamos implementar una solución top-down recursiva (Memoization):
    \begin{itemize}
       \item Si para calcular un $best(i,j)$ necesitamos un resultado ya calculado, lo usamos directamente.
       \item Sino, lo calculamos recursivamente, almacenamos su valor en la tabla de resultados y luego lo utilizamos.
    \end{itemize}
     
    
\end{frame}


\begin{frame}[fragile]{Algoritmo bottom-up}
\begin{lstlisting}
longitudCaminoMinimo(Matriz, m, n):
    best <- matrizDeEnteros(m,n)
    best[m,n] = Matriz[m,n]
    FOR i <- m-1 DOWNTO 1 DO
        best[i,n] = Matriz[i,n] + best[i+1,n]
    FOR j <- n-1 DOWNTO 1 DO
        best[m,j] = Matriz[m,j] + best[m,j+1]
    FOR i <- m-1 DOWNTO 1 DO
        FOR j <- n-1 DOWNTO 1 DO
            best[i,j] = Matriz[i,j] + min(best[i+1,j] + best[i,j+1])
    RETURN best[1,1]
\end{lstlisting}

La complejidad del algoritmo resultante es $O(nm)$, tanto espacial como temporal. Se puede bajar la complejidad espacial a $O(min(n,m))$ si no interesa reconstruir el camino sino solo su longitud.
\end{frame}
